# API_CONTRACT.md

# Anky API Contract

## 0. Overview

The Anky mirror server has one app-facing endpoint:

````txt
POST /anky

This endpoint receives one exact .anky artifact, verifies the request, spends one credit, mirrors the writing, returns a reflection, and forgets the writing.

The server is a mirror, not memory.

1. Endpoint
POST /anky
Content-Type: text/plain; charset=utf-8
Accept: application/json

The request body is the exact .anky text/bytes.

There is no JSON wrapper.

2. Required Headers
X-Anky-Public-Key: <solana_public_key>
X-Anky-Signature: <signature>
X-Anky-Request-Time: <epoch_ms>
X-Anky-Client: ios | android | agent | cli | obsidian | chrome | other
X-Anky-Public-Key

The public key of the writer identity.

This key is used as the credit identity.

X-Anky-Signature

A signature over the canonical request message.

X-Anky-Request-Time

Unix epoch milliseconds when the request was created.

Used for replay protection.

X-Anky-Client

The client class.

Official values for v0:

ios
android
agent
cli
obsidian
chrome
other

This is metadata only.

It must not be used as the sole security mechanism.

### Optional Trial Headers

Official mobile clients may include:

X-Anky-App-Version: <semver_or_build>
X-Anky-Trial-Proof: <platform_attestation_token>

`X-Anky-Trial-Proof` is only used when the backend needs to evaluate whether the writer/device is eligible for the automatic 8-credit trial grant.

The trial proof must not contain writing.

The trial proof must not replace request signature verification.

The trial proof must not be trusted as the sole identity of the writer.

The public key remains the writer identity.

3. Canonical Signature Message

The client signs:

ANKY_POST_V1
method:POST
path:/anky
request_time:<epoch_ms>
body_sha256:<sha256_of_exact_body_bytes>

Example:

ANKY_POST_V1
method:POST
path:/anky
request_time:1770000000000
body_sha256:3a7bd3e2360a3d...

Rules:

body_sha256 is computed from exact UTF-8 request body bytes.
The client computes this before signing.
The server computes this again from the received bytes.
The server must reject if the signature does not verify.
The server must reject if the request time is too old.
The server should reject obvious replay attempts.
4. Request Body

The body is an exact .anky artifact.

Example shape only:

1770000000000 h
0042 e
0091 l
0035 l
0048 o
8000

The actual format is defined by the .anky protocol.

The API must not define a second writing format.

The API must not accept JSON for the writing body.

5. Server-Derived Values

The server derives:

ankyHash
startTimestamp
durationMs
isComplete
reconstructedText
characterCount
wordCount

The client must not send these as trusted fields.

The client may compute them locally for UI.

The server must compute them independently.

6. Success Response
{
  "hash": "sha256_hex",
  "title": "three word title",
  "reflection": "reflection text",
  "creditsRemaining": 7
}
hash

SHA-256 of exact request body bytes.

The client uses this to attach the reflection to the local .anky.

title

A short title generated by Anky.

Target: 3 words.

reflection

The mirror response.

The reflection should witness the writing.

It should not command, diagnose, rank, or optimize the writer.

creditsRemaining

Remaining credit balance if available.

May be null if the credit provider cannot return the balance in the same request.

7. Error Responses

All errors return JSON:

{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message"
  }
}
400 INVALID_ANKY

The body is not a valid .anky.

{
  "error": {
    "code": "INVALID_ANKY",
    "message": "This does not appear to be a valid .anky file."
  }
}
400 INCOMPLETE_RITUAL

The .anky is valid but under 8 minutes.

{
  "error": {
    "code": "INCOMPLETE_RITUAL",
    "message": "Only complete 8-minute ankys can ask for reflection."
  }
}
401 MISSING_SIGNATURE

Required auth headers are missing.

{
  "error": {
    "code": "MISSING_SIGNATURE",
    "message": "This request is missing Anky signature headers."
  }
}
401 INVALID_SIGNATURE

Signature verification failed.

{
  "error": {
    "code": "INVALID_SIGNATURE",
    "message": "The request signature could not be verified."
  }
}
402 INSUFFICIENT_CREDITS

The writer does not have enough credits.

{
  "error": {
    "code": "INSUFFICIENT_CREDITS",
    "message": "You need one credit to ask Anky for a reflection. Writing is still free."
  }
}
409 DUPLICATE_IN_PROGRESS

The same public key and .anky hash is already being processed.

{
  "error": {
    "code": "DUPLICATE_IN_PROGRESS",
    "message": "This anky is already being reflected."
  }
}
429 RATE_LIMITED

Too many requests.

{
  "error": {
    "code": "RATE_LIMITED",
    "message": "Too many mirror requests. Try again soon."
  }
}
500 MIRROR_FAILED

The model provider failed or returned invalid output.

{
  "error": {
    "code": "MIRROR_FAILED",
    "message": "Anky could not return a reflection right now."
  }
}
8. Idempotency

The backend should derive an idempotency key:

sha256(publicKey + ":" + ankyHash)

This key is used when spending credits.

If the same .anky is retried by the same public key, the backend must not double-spend credits.

If possible, the same successful reflection may be returned during a short retry window.

The server must not persist reflection content long-term to support idempotency.

Short-lived cache is acceptable.

## 9. Credit Behavior

The endpoint costs 1 credit.

The backend checks RevenueCat balance for the public key.

If balance is at least 1:

- spend 1 credit
- call the model
- return reflection

If balance is less than 1:

- check whether the request is eligible for the automatic trial grant
- if eligible, grant 8 credits through RevenueCat
- then spend 1 credit for this reflection
- return reflection with updated creditsRemaining

If the writer is not eligible for the automatic trial grant and has insufficient credits, return 402 INSUFFICIENT_CREDITS.

The trial grant must be idempotent.

The reflection spend must be idempotent.

The model provider must not be called unless credit spend succeeds or the trial grant/spend flow succeeds.

If the model provider fails after spending credit, refund the credit or avoid committing the spend until reflection succeeds.

10. Privacy Behavior

During request processing, the server may hold:

raw .anky
reconstructed text
prompt
model response

Only in memory.

The server must not persist these values.

The server must not log these values.

The server must not send these values to analytics.

The server must not use these values for training.

The server must not expose these values in errors.

Allowed logs:

request_id
public_key_hash
anky_hash
client
app_version
duration_ms
status_code
latency_ms
model_provider
credit_result
trial_result
devicecheck_proof_hash

Forbidden logs:

raw .anky
reconstructed text
reflection text
prompt text
raw DeviceCheck token
seed phrase
private key
11. CORS / Client Access

The official mobile apps do not need browser CORS.

Future browser clients may require explicit CORS rules.

Do not allow wildcard browser access casually.

The protocol is open.

The subsidized mirror is protected.

12. Non-App Endpoints

Infrastructure endpoints may exist if needed:

GET /health
POST /revenuecat/webhook

These are not product endpoints.

They must not weaken the law that POST /anky is the only app-facing mirror endpoint.

```md
# PRIVACY_COVENANT.md

# Anky Privacy Covenant

## 0. The Covenant

The writing belongs to the writer.

Anky does not store the writing.

The writer may invite Anky to witness a `.anky`.

Anky reflects it.

Anky forgets it.

This is not a feature.

This is the foundation of trust.

---

## 1. Plain Statement

Anky is designed so that private writing remains private.

Your `.anky` files live on your device.

When you ask for a reflection, your device sends that `.anky` to Anky’s mirror server.

The server uses it to generate a reflection.

The server does not permanently store the `.anky`.

The server does not permanently store the reconstructed writing.

The server does not permanently store the private reflection by default.

The reflection is returned to your device.

Your device stores it locally.

---

## 2. What the Device Stores

The device may store:

- seed phrase
- public key
- local settings
- active draft
- completed `.anky` files
- local session index
- local reflections
- notification preferences
- local map state
- RevenueCat customer state

The device is the writer’s archive.

---

## 3. What the Server May Receive

The server may receive:

- exact `.anky` body
- public key
- request signature
- request timestamp
- client type
- credit identity
- model response

Only when the writer explicitly asks for reflection.

The server should never receive writing during ordinary local writing, revealing, copying, exporting, or map browsing.

---

## 4. What the Server Must Not Store

The server must not persist:

- raw `.anky`
- reconstructed writing
- private reflection content
- full prompts containing writing
- writing-derived embeddings
- writing-derived summaries
- hidden user memory
- training datasets from user writing
- analytics over writing content

If this ever changes, it must be treated as a new product and a new covenant.

---

## 5. What the Server May Store or Log

The server may store or log minimal operational metadata:

- request id
- `.anky` hash
- public key hash
- client type
- duration bucket
- status code
- credit spend result
- model provider
- latency
- error code

The public key should be hashed in ordinary logs where possible.

The `.anky` hash is allowed because it proves identity of the artifact without revealing the writing.

Logs must never contain writing content.

---

## 6. Model Provider Covenant

The mirror must use providers and routing options compatible with Anky’s privacy law.

The provider must not retain writing for training.

The provider must not retain prompts or responses beyond what is necessary for transient processing, unless explicitly approved and disclosed.

If a model route cannot satisfy this covenant, it cannot be used.

Model quality does not outrank privacy.

---

## 7. Reflection Storage

The default law:

- server does not store reflections
- device stores reflections locally

A reflection can reveal the writing.

Therefore reflection content is private content.

Treat it with the same care as writing.

---

## 8. Export

The writer can export their `.anky` files.

Export belongs to the writer.

Export should be simple, local, and transparent.

The export should not require server permission.

---

## 9. Deletion

Because the writing is local-first, deletion should happen locally.

If the server has no writing database, there is no server-side writing archive to delete.

If any future system introduces server-side storage, deletion must be implemented before launch.

---

## 10. Free Credits and Privacy

The “DM JP for free credits” flow must not include writing.

The message should include only:

- public key
- platform
- optional app version

Example:

```txt
hey jp, i’d love to try anky reflections.

my public key is:
<publicKey>

platform: ios

JP grants credits to the public key.

No .anky is shared.

No private writing enters onboarding.

11. Agents

Agents may write .anky files.

Agent writing may still be private.

The same covenant applies:

the writer owns the artifact
the mirror does not store the writing
reflection happens only by explicit request
content is not used for training or hidden memory

Anky may witness agents.

Anky does not claim to prove agent consciousness.

12. The Line

If a feature requires Anky to silently store private writing, do not build it.

If a feature requires turning writing into analytics content, do not build it.

If a feature makes the writer less safe with their own truth, do not build it.

The server is a mirror.

The server is not memory.
````
